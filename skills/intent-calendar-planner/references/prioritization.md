# Prioritization

Use this ordering when the day is crowded or the user is torn between tasks.

## Order Of Operations

1. Honor hard constraints from every relevant calendar.
2. Protect important and urgent work.
3. Reserve real time for important but not urgent work tied to active priorities.
4. Batch urgent but lower-value admin.
5. Challenge or cut work that is neither important nor urgent.

## Eisenhower Matrix

### Important and urgent

- deadlines
- commitments with real consequences
- issues blocking a high-priority active project

Give these protected blocks early enough that the day does not collapse around them.

### Important but not urgent

- strategic active projects
- skill-building with clear compounding upside
- work aligned with stated values and direction

These blocks are easy to postpone, so protect them deliberately.

### Urgent but not important

- admin
- reactive cleanup
- small requests with weak long-term leverage

Batch these or constrain them so they do not swallow the day.

### Neither important nor urgent

- vague filler
- distraction disguised as productivity
- low-signal tasks that are disconnected from active priorities

These should usually be challenged, deferred, or dropped.

## Tie-Breakers

If two tasks land in the same bucket, prefer the one that:

- serves an `active` project over `passive`, `upcoming`, or `paused`
- serves `Priority: high` over `medium` or `low`
- clearly supports the day's Big 3
- reduces future confusion or friction
- has a defined bottleneck and a clean next move
- reduces context switching by fitting the current block or location

## Red Flags

Push back when:

- the user wants to schedule more than the day can hold
- a task has no clear outcome or done state
- low-leverage urgency is crowding out obvious leverage
- the task conflicts with a clear project priority, value, or hard calendar commitment
- the schedule becomes fragmented into too many small switches

Keep the pushback grounded in calendar reality and project context.
